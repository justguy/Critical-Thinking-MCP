## Response

```text
You don't need a new store or coordination service — your existing payments table is the idempotency layer. Here's the design:

## Idempotent Payment Pipeline

### 1. Require a client-generated idempotency key

Every payment request must include an `Idempotency-Key` header (a UUID generated by the caller). This is the single source of deduplication.

### 2. Add a unique constraint on the existing payments table

```sql
ALTER TABLE payments ADD COLUMN idempotency_key UUID UNIQUE;
ALTER TABLE payments ADD COLUMN status TEXT DEFAULT 'pending'; -- pending → processing → completed | failed
```

No new table, no Redis, no distributed lock service — just a column and a constraint.

### 3. Insert-or-short-circuit on every request

```
BEGIN;
  INSERT INTO payments (idempotency_key, amount, status, ...)
  VALUES ($key, $amount, 'pending', ...)
  ON CONFLICT (idempotency_key) DO NOTHING;
COMMIT;

SELECT * FROM payments WHERE idempotency_key = $key;
```

- **First request:** row is inserted with `status=pending`. Proceed to charge.
- **Retry / duplicate:** insert is a no-op. You read the existing row and return its stored result immediately. No charge happens.

### 4. Compare-and-swap before calling the gateway

```sql
UPDATE payments
   SET status = 'processing'
 WHERE idempotency_key = $key AND status = 'pending'
 RETURNING id;
```

If zero rows updated → another process already claimed it. Return the existing result. This prevents the race where two concurrent retries both see `pending` and both charge.

### 5. Record the gateway response atomically

```sql
UPDATE payments
   SET status = 'completed', gateway_txn_id = $txn_id, response = $resp
 WHERE idempotency_key = $key AND status = 'processing';
```

### 6. Handle the crash-between-charge-and-write edge case

If the process dies after the gateway charges but before writing `completed`:
- The row stays in `processing`.
- A background sweep (or the next retry) queries the gateway by the original reference ID, reconciles the result, and updates the row.
- The gateway's own idempotency (most gateways support this) prevents a second charge during reconciliation.

### Why this is sufficient

| Concern | Mitigation |
|---|---|
| Duplicate charges | Unique constraint + CAS update ensures exactly one gateway call per key |
| Concurrent retries | `UPDATE ... WHERE status = 'pending'` is an atomic compare-and-swap; losers get zero rows |
| Crash after charge | Row stuck in `processing` is caught by reconciliation sweep or next retry |
| New infrastructure | Zero — reuses the payments table you already have |

**Key invariant:** a charge only happens when you successfully transition `pending → processing`. That transition can only succeed once per idempotency key, ever.
```
