# When "Valid" Doesn't Mean "Ready"

> AI said: "This plan is valid."
> Reality: it would fail on the first run.

This is a real AI-generated execution plan. 31 steps, 20+ shared resources, complex dependency chains. Every validation check passed. The logic was sound. The dependencies were correct.

It still wasn't safe to execute.

Here's how we know — and why most systems would miss it.

---

## The Plan

A 31-step execution plan for building a multi-agent software system:

- **4 infrastructure phases** (P0-P4): sandbox isolation, timeout scaling, interface registry, session persistence
- **3 capability tracks** running in parallel:
  - Track A: risk assessment, parallel execution, coherence checking
  - Track B: pattern learning from failures
  - Track C: adaptive decomposition when plans stall
- **1 scorecard step** (SC): metrics aggregation across all tracks
- **3 validation experiments** (V1-V3): progressively larger real-world tests

20+ shared resources. Complex dependency chains. Generated by an AI planning agent.

On paper, it looks solid. Three independent checks agree.

---

## Every Check Passed

Three tools were run. Each one said: looks good.

### Reasoning Structure (`validate_reasoning_chain`)

**Result: PASS** (grounding score: 0.733)

- No circular reasoning detected
- No orphaned conclusions (every recommendation traces back to evidence)
- 26 nodes, 24 edges — well-connected graph

**What this means:** The plan's logic is internally consistent. Claims lead to evidence, evidence supports conclusions. Nothing contradicts itself.

### Plan Validity (`check_plan_validity`)

**Result: PASS** (completeness: 0.80) — but with 7 warnings

- No circular dependencies
- No missing prerequisites
- Critical path: 5 steps through risk assessment → scorecard → validation

But the tool found **7 resource conflicts**:

| Shared Resource | Steps That Modify It | Risk |
|---|---|---|
| Core execution module | 3 steps across different phases | Merge conflict if phases overlap |
| Verification helpers | 5 steps across 3 tracks | Most contended — 5 independent writers |
| Planning module | 2 steps in different tracks | Design collision risk |
| Retry handler | 2 steps in different tracks | Behavioral override risk |
| Decision log | 2 steps in different tracks | Schema conflict risk |
| Task tracker | 2 steps in different tracks | State mutation conflict |
| Validation pipeline | 3 sequential experiments | Expected — experiments are ordered |

**What this means:** The plan is structurally valid — no impossible dependencies, no cycles. But 6 of 7 resource conflicts are real: multiple steps modify the same module without enforced ordering. If any phases run concurrently (or if an agent doesn't commit between phases), changes will collide.

### Response Quality (`score_response_quality`)

**Result: PASS** (overall: 0.61)

| Dimension | Score | Assessment |
|---|---|---|
| Substance | 0.969 | Strong — dense, diverse vocabulary |
| Specificity | 0.179 | **Weak** — describes intent, not implementation |
| Hedge density | 0.000 | No hedging — direct and confident |
| Structure | 0.330 | Partial — has claims but evidence chain is implicit |

**What this means:** The plan is substantive and confident — but underspecified. It says *what* to build, not *how*. An executing agent would need to make many implementation decisions the plan doesn't address.

---

## So What's the Problem?

Each tool passed individually. But look at what happens when you combine them:

| Check | Individual Result | Combined Insight |
|---|---|---|
| Reasoning | PASS (0.733 grounding) | Logic is sound |
| Structure | PASS (0.80 completeness) | Dependencies are valid |
| Quality | PASS (0.61 overall) | Content is substantive |
| **Execution readiness** | **NOT ASSESSED by any single tool** | **7 resource conflicts + 0.18 specificity = execution risk** |

No single tool flagged this plan as unsafe. But the combination says: **don't run this yet.**

### What would actually break

Two phases modify the same core module. Neither knows about the other. The later phase overwrites earlier changes. No mechanism detects the overwrite.

**Result:** Silent logic loss. The system behaves incorrectly — but nothing errors out. No stack trace. No failed test. Just wrong behavior in production.

This isn't hypothetical. It's one of the most common failure modes in multi-agent systems. And most planning tools have no way to catch it.

---

## How Confident Should You Be?

The plan's honest confidence ceiling: **0.37 out of 1.0**.

That's low. Here's why:

- The weakest assumption — whether a cold-start agent can execute these steps — had confidence 0.70 but no way to falsify it without actually running it
- Several integration assumptions can't be tested until the validation experiments complete
- The plan knows this: V1-V3 are designed to confirm or kill these assumptions

The plan doesn't overclaim. That's good. But 0.37 means: *don't bet production on this until V1-V3 produce real data.*

---

## How Does This Actually Work?

The detection is surprisingly simple:

1. For each step: what does it read? What does it write?
2. Group steps by shared resources
3. Do ordering constraints exist between writers?

If multiple steps write to the same resource and nothing enforces who goes first — that's a resource conflict. No AI needed. Just graph analysis.

---

## What Would Fix This Plan?

Four changes:

1. **Add ordering constraints for shared resources.** The 6 real resource conflicts need explicit dependency edges or "commit after phase" gates.

2. **Make steps specific enough to execute.** Specificity score of 0.18 means the executing agent will face ambiguity at every step. Each one needs: inputs, outputs, success condition, rollback path.

3. **Don't claim confidence you haven't earned.** The 0.37 ceiling is honest — keep it. Don't raise it until V1-V3 produce real data.

4. **Commit between phases.** The implementing agent should checkpoint after each phase. This is an operational requirement the plan never states — and the most likely source of silent failures.

---

## Why Don't Other Systems Catch This?

Most AI planning systems ask:
- Does the plan make sense? *(plausibility)*
- Are all steps present? *(completeness)*
- Do dependencies work? *(consistency)*

They stop there. They don't ask:
- Will parallel tracks collide on shared resources?
- Are steps specific enough to execute without guessing?
- Are confidence claims backed by testable assumptions?
- Is this plan actually *safe to run*?

Ask an LLM to review this plan and it will say:

> "The plan is complete and logically sound. Ready to execute."

CT-MCP says:

> "The plan is valid — but will likely fail due to unresolved execution constraints."

Same plan. Opposite conclusions. The difference isn't better AI — it's a different kind of check entirely. Deterministic math, not judgment calls.

## What It Catches — and What It Doesn't

**Caught:**
- 7 resource conflicts (6 real risks, 1 expected)
- Low specificity (0.18) — the plan says *what* but not *how*
- Confidence ceiling (0.37) — driven by assumptions you can't test yet
- The gap between "valid" and "executable"

**Not caught:**
- Whether the component designs are technically sound *(needs domain expertise)*
- Whether the time estimates are realistic *(needs project context)*
- Whether the executing agent will guess correctly on ambiguous steps *(needs runtime observation)*

These are real boundaries. The tool catches structural and mathematical flaws. It doesn't replace your judgment.

Every AI planning system today asks: *"Does this plan make sense?"*

Almost none of them ask: *"Will this plan survive execution?"*

That's the gap. A plan can pass every check — reasoning, structure, quality — and still break the moment you run it.
